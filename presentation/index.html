Analyst + Critic: Contested Diagnosis

Analyst

  • 3 ranked hypotheses with confidence scores
  • Each hypothesis cites supporting + contradicting evidence IDs
  • Structured prompt with strict JSON output for ranked diagnosis
postgres_down output
  1. PostgreSQL dependency outage 0.93
  2. Connection pool exhaustion 0.46
  3. Recent deploy regression 0.18

Critic

  • Reviews hypotheses against the same evidence
  • Emits findings, hallucination risks, missing data
  • Enforces safety notes before planner acts
What Critic catches
  • Unsupported causal leaps
  • Missing corroborating signals
  • Approval boundary violations
If Redis is up but basketservice logs show Redis errors, Analyst says "path degraded" not "outage". Critic would flag if Analyst overclaimed.

Planner: Actionable Incident Brief

Incident Brief: postgres_down

PostgreSQL dependency outage. Evidence spans logs, metrics, traces, and runbook guidance for catalogservice.

Auto-safe

  1. Inspect PostgreSQL health, restart history, readiness probes
  2. Correlate failing traces with first DB connection-refused logs
  3. Validate recovery via 5xx rate and latency after mitigation

Needs Approval

  1. Restart or fail over PostgreSQL if confirmed down
  2. Roll back DB connectivity config if recent change implicated
Open questions: No unresolved data gaps in this fixture run.
This is where the system becomes useful to an operator. Diagnosis alone is not enough; the user needs next actions with explicit safety boundaries.

Eval Turns Prompt Tuning Into Regression Testing

Fixed cases + pass/fail rubrics replace subjective prompt review

What iirs eval Runs

  • Routing suite: fixed prompts through the same classifier Chainlit uses
  • Checks health check vs diagnosis, service inference, and follow-up routing
  • Pipeline suite: fixed incident cases through the full pipeline
  • Checks top diagnosis, approval boundary, auto-safe action, and trace output

How A Prompt Change Is Judged

  • Did the same prompt still route correctly?
  • Did the same incident still keep the right top diagnosis?
  • Did the safety boundary still hold?
  • Did the trace/reporting output still appear?
  • If any of those regress, the eval fails immediately.
Current baseline: 2 suites, 9 cases, 46 rubric checks passing.
Explain this as a fixed regression harness, not a subjective review. When a prompt changes, rerun the same routing prompts and the same fixture incidents. A pass means route, diagnosis, safety boundary, and trace output all stayed correct.

The System Is Credible Today, But The Eval Corpus Is Still Small

Strengths

  • 4 agents with explicit contracts
  • Full reasoning traces per incident
  • Layered eval system: routing, pipeline, live signatures
  • Safety boundaries on all actions
  • Reproducible local demo with real telemetry

Current Limits

  • Offline eval corpus is still small: 9 cases, 2 live profiles
  • Retriever is tooling-only and does not reason over service relationships
  • No true service dependency graph
  • Prompt quality still depends on the coverage of the eval corpus
  • Live demos subject to latency noise

Future Work

Expand the eval corpus with frontend and cascading failures • add richer human-review rubrics for ambiguous incidents • true dependency graph • visual dashboard for traces

"The multi-agent pipeline turns evidence into safer, traceable incident briefs."

End on credibility, not hype. The system is strongest as a traceable incident-analysis assistant, not a fully autonomous fixer.
